# Эксплуатация и реагирование

## Перед каждым production deployment

1. Выполнить все новые D1-миграции по порядку.
2. Проверить обязательный binding `DB`; если R2 доступен — также `MEDIA`, иначе проверить D1 fallback изображений.
3. Убедиться, что секреты заданы только в Cloudflare, а не в GitHub.
4. Выполнить `pnpm install --frozen-lockfile` и `pnpm run check` (полный аудит включает dev-зависимости).
5. Проверить регистрацию, письмо, вход, выход, восстановление, обращение, загрузку изображения и вход администратора.
6. Проверить приёмную: приватную категорию, премодерацию, официальный ответ, отметку интереса и аудируемое раскрытие автора.
7. Проверить заголовки CSP/HSTS и отсутствие ошибок в Functions logs.

## Проверка состава организации

1. Применить `0005_staff_roles_presence.sql` до публикации кода с должностями.
2. Назначить первый `Owner` в админ-панели → «Состав и должности».
3. Проверить порядковый ID в профиле и сохранение назначенных должностей.
4. Убедиться, что публичная панель сотрудников не раскрывает email, внутренние UUID, IP и точное время активности.
5. Проверить добровольное скрытие статуса сотрудником и запись назначений в `admin_audit_log`.

## Обязательные секреты

- `ADMIN_TOKEN`: уникальная случайная парольная фраза не короче 20 символов;
- `RATE_LIMIT_SECRET`: отдельный случайный секрет не короче 32 байт;
- `CODE_PEPPER`: ещё один отдельный случайный секрет не короче 32 байт;
- `RESEND_API_KEY`, `EMAIL_FROM`;
- `TURNSTILE_SECRET_KEY` и build-переменная `VITE_TURNSTILE_SITE_KEY`.

Рекомендуется `ADMIN_TOTP_SECRET` — Base32-ключ, вручную добавленный в приложение-аутентификатор. Серверный и публичный ключи Turnstile включаются одновременно; отсутствие серверного ключа в production блокирует защищённые формы.

## Cloudflare

- SSL/TLS: Full (strict), Always Use HTTPS, minimum TLS 1.2;
- DNS-записи сайта: Proxied;
- Bot Fight Mode и Browser Integrity Check: включены;
- Turnstile: hostname `aogd.site` и `www.aogd.site`;
- секреты: одинаково настроены для Production, а для Preview — только если там нужен backend;
- D1 и, если используется, R2: регулярные экспорт/резервная копия и проверка восстановления.

## Ротация

- `ADMIN_TOKEN`: после смены удалить строки из `admin_sessions`, затем сделать новый deployment;
- `ADMIN_TOTP_SECRET`: заменить ключ в аутентификаторе и Cloudflare одновременно;
- `RATE_LIMIT_SECRET`: можно менять независимо; старые rate-limit ключи станут недоступны;
- `CODE_PEPPER`: не менять без плана, пока существуют активные email-коды;
- `RESEND_API_KEY`: отозвать старый ключ после проверки нового.

## Если подозревается взлом

1. Для блокировки трафика использовать Cloudflare WAF/Under Attack/Access rule; frontend maintenance mode меняет только интерфейс и не закрывает `/api/*`.
2. Отозвать сессии: очистить `admin_sessions` и при необходимости `user_sessions`.
3. Сменить `ADMIN_TOKEN`, TOTP, Resend API key, GitHub и Cloudflare credentials.
4. Сохранить Functions/Cloudflare/GitHub audit logs и время инцидента.
5. Проверить `admin_audit_log`, новые записи D1 и объекты R2.
6. Восстановить чистую версию, проверить миграции и только затем открыть сайт.
7. Если затронуты пользовательские данные, уведомить пользователей и выполнить требования применимого закона.

Гостевые обращения и вложения автоматически удаляются после 180 дней при очередном цикле очистки. Для строгого соблюдения более короткого срока хранения настройте отдельную регулярную D1-задачу и проверяйте её журнал выполнения.
