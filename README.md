# A.O.G.D Records

Сайт A.O.G.D с публичным реестром, личными кабинетами, поддержкой и закрытой административной панелью.

## Что работает

- публичные записи, поиск и подробные карточки;
- регистрация по почте и паролю;
- подтверждение почты шестизначным кодом;
- восстановление пароля через почту;
- профиль участника и история его обращений;
- постоянный порядковый ID каждого аккаунта;
- защищённые должности сотрудников, пользовательские касты и публичный статус присутствия без раскрытия IP;
- обращения в поддержку с Telegram username и изображением;
- «Приёмная A.O.G.D» с премодерацией публичных вопросов и официальными ответами;
- приватные жалобы, исправления и сообщения о безопасности;
- псевдоанонимные публикации: посетители не видят автора, а служебное раскрытие личности требует причины и записывается в аудит;
- отдельная обработка обращений в административной панели;
- таблица лидеров по количеству одобренных обращений;
- гости могут обращаться в поддержку, но не попадают в рейтинг;
- светлая, тёмная и синяя темы, русский и английский интерфейс.

Профиль и поддержка: `/#/profile`.

Публичная приёмная: `/#/reception`.

Закрытая административная панель: `/#/aogd-vault-7m4k9p`.

## Локальный запуск

Нужен Node.js 22 или новее (Wrangler 4.x требует Node 22).

```bash
pnpm install
pnpm dev
```

Обычный Vite-сервер показывает только интерфейс. Для локальной проверки Cloudflare Pages Functions используйте:

```bash
pnpm db:local:apply
pnpm dev:cloudflare
```

Первая команда применит все D1-миграции к отдельной локальной базе. Вторая соберёт сайт и поднимет Pages runtime с локальными bindings `DB` и `MEDIA` на `http://localhost:8788`. Email и Turnstile требуют тестовых секретов в `.dev.vars` — не добавляйте этот файл в Git. `wrangler.local.jsonc` предназначен только для разработки и не содержит идентификаторов production-ресурсов.

## Публикация Cloudflare Pages

Подключите этот GitHub-репозиторий к Cloudflare Pages и укажите:

- Framework preset: `React (Vite)`;
- Build command: `pnpm build`;
- Build output directory: `dist`;
- Root directory: `/`.

Файл `public/_routes.json` уже направляет все `/api/*` в Pages Functions.

### 1. База D1

В существующей базе D1 последовательно выполните в SQL Console:

1. `migrations/0001_init.sql` — публичные записи;
2. `migrations/0002_accounts_support.sql` — аккаунты, коды, сессии и обращения;
3. `migrations/0003_security_hardening.sql` — защищённые админ-сессии, rate limit, аудит, блокировка аккаунтов и усиленные хеши паролей;
4. `migrations/0004_reception.sql` — вопросы приёмной, официальные ответы и отметки интереса;
5. `migrations/0005_staff_roles_presence.sql` — порядковые ID аккаунтов, должности, назначения и приватный статус присутствия сотрудников;
6. `migrations/0006_privacy_and_owner_invariants.sql` — повторный opt-in статуса присутствия и запрет удаления последнего Owner.

В настройках Pages должна существовать привязка:

- Variable name: `DB`;
- D1 database: ваша база, например `aogd-records-db`.

Если Cloudflare пишет, что переменная `DB` уже существует, ничего добавлять не нужно: привязка уже настроена.

### 2. R2 для фотографий публичного реестра

Если R2 доступен, добавьте binding:

- Variable name: `MEDIA`;
- R2 bucket: ваш закрытый bucket.

Если R2 недоступен, не создавайте фиктивный binding `MEDIA`: приложение сохранит проверенное изображение размером до 900 КБ непосредственно в D1 и будет отдавать его отдельным ограниченным API-маршрутом. В таком режиме статус `mediaStorage` остаётся выключенным — это ожидаемое ограничение, а не утечка секрета.

Фотографии обращений поддержки автоматически уменьшаются до 700 КБ и хранятся в D1. Они не публикуются в таблице лидеров.

### 3. Переменные и секреты

В Cloudflare Pages откройте **Settings → Variables and Secrets** и добавьте в Production:

| Имя | Тип | Значение |
| --- | --- | --- |
| `ADMIN_TOKEN` | Secret | отдельный длинный пароль администратора |
| `ADMIN_TOTP_SECRET` | Secret | необязательный Base32-ключ 2FA из приложения-аутентификатора |
| `RATE_LIMIT_SECRET` | Secret | отдельный случайный секрет для обезличенных лимитов |
| `CODE_PEPPER` | Secret | отдельный случайный секрет для email-кодов |
| `TURNSTILE_SECRET_KEY` | Secret | серверный ключ Cloudflare Turnstile |
| `RESEND_API_KEY` | Secret | API-ключ Resend |
| `EMAIL_FROM` | Text | например `A.O.G.D <support@updates.aogd.site>` |
| `SUPPORT_EMAIL` | Text | почта, на которую приходят уведомления о новых обращениях |

`RESEND_API_KEY`, пароли и токены нельзя загружать в GitHub.

В разделе build-переменных добавьте `VITE_TURNSTILE_SITE_KEY` — публичный Site key от того же Turnstile widget. В production серверный и публичный ключи обязательны и включаются одновременно; backend намеренно блокирует защищённые формы, если серверный ключ отсутствует. Без ключей допускается только локальная разработка на localhost.

### 4. Настройка писем

1. Создайте аккаунт в Resend.
2. Добавьте и подтвердите домен отправителя. Удобнее использовать поддомен, например `updates.aogd.site`.
3. Добавьте предложенные Resend DNS-записи в Cloudflare DNS.
4. Создайте API key и сохраните его как секрет `RESEND_API_KEY`.
5. Укажите адрес с подтверждённого домена в `EMAIL_FROM`.

Без этих трёх настроек регистрация с кодом и восстановление пароля не смогут отправлять письма.

### 5. Новый deployment

После миграции и добавления переменных запустите новый Production deployment. Старый deployment не получает добавленные позже переменные автоматически.

## Как формируется рейтинг

- Обращение создаётся со статусом `pending`.
- Администратор открывает вкладку **Обращения поддержки**.
- Статус `approved` добавляет один балл подтверждённому участнику.
- `rejected` и `resolved` баллов не дают.
- Повторная смена статуса не создаёт дополнительных баллов: рейтинг считает текущее число одобренных обращений.
- Email, Telegram username, текст обращения и фото в публичный рейтинг не передаются.

## Почему GitHub Pages отключён

GitHub Pages публикует только статические файлы и не поддерживает этот backend. Отдельный workflow GitHub Pages намеренно отсутствует, чтобы не создавать устаревшую копию сайта и конфликт за основной домен. Production должен открываться с Cloudflare Pages либо с другого хостинга, поддерживающего серверный API и базу.

## Безопасность

- Пароли хешируются PBKDF2-SHA256 с индивидуальной солью и 600 000 итерациями; старые хеши усиливаются после успешного входа.
- Пользовательские и административные сеансы хранятся на сервере; в браузере остаются только cookies `HttpOnly`, `Secure`, `SameSite`.
- Коды хранятся как HMAC, действуют 15 минут и ограничены шестью попытками.
- Административный пароль не хранится в браузере и после входа заменяется восьмичасовой серверной сессией; поддерживается TOTP-2FA.
- Формы защищены проверкой origin, лимитами, ограничением размера, Turnstile и проверкой содержимого изображений.
- Ошибки API не раскрывают D1, SQL, секреты или внутренние пути; административные изменения попадают в обезличенный журнал.
- Публичная приёмная не показывает email или идентификатор автора. Анонимный режим скрывает никнейм только от посетителей, а не от администрации; каждый служебный просмотр автора требует обоснования и фиксируется в журнале.
- Жалобы, исправления данных и сообщения об уязвимостях принудительно остаются приватными на уровне API и ограничений D1.
- В рейтинге видны только никнейм и число одобренных обращений.
- Не принимайте личную, заведомо недостоверную или полученную незаконным способом информацию, документы, домашние адреса, пароли и платёжные данные.
